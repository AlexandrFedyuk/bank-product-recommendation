# Мониторинг рекомендательной системы

Мониторинг нужен, чтобы контролировать качество модели и стабильность сервиса после выкатки.

## 1. Технические метрики

Эти метрики помогают понять, работает ли API стабильно.

| Метрика | Что показывает | Где используется |
|---|---|---|
| `recommendation_requests_total` | количество запросов к `/recommend` | контроль нагрузки |
| `recommendation_errors_total` | количество ошибок API | алерты при росте ошибок |
| `recommendation_latency_seconds` | время ответа API | контроль SLA |

## 2. Модельные метрики

В офлайн-валидации считаются:

| Метрика | Зачем нужна |
|---|---|
| `MAP@7` | основная ranking-метрика качества |
| `Precision@7` | точность списка рекомендаций |
| `Recall@7` | полнота найденных новых покупок |
| `HitRate@7` | доля клиентов с хотя бы одной удачной рекомендацией |
| `Coverage@7` | разнообразие рекомендуемых продуктов |

Эти метрики логируются в MLflow при запуске `scripts/train.py`.

## 3. Метрики данных

В production-среде нужно отслеживать:

- долю пропусков в `age`, `renta`, `segmento`, `canal_entrada`;
- долю неизвестных категорий;
- распределение возраста;
- распределение дохода;
- долю активных клиентов;
- среднее количество продуктов на клиента;
- изменение частот продуктовых признаков.

Если распределения сильно изменились относительно обучающей выборки, это может означать data drift.

## 4. Метрики рекомендаций

API отправляет счётчик:

```text
recommended_product_total{product="..."}
```

Он показывает, какие продукты чаще всего попадают в рекомендации. Это помогает обнаружить ситуацию, когда модель начала рекомендовать слишком узкий набор продуктов.

Важно отслеживать:

- топ рекомендуемых продуктов;
- долю каждого продукта в рекомендациях;
- coverage;
- долю клиентов без рекомендаций;
- средний score модели, если scores сохраняются в логах.

## 5. Бизнес-метрики

После интеграции с продуктовым контуром нужно отслеживать:

| Метрика | Смысл |
|---|---|
| CTR рекомендации | клиент кликнул на предложение |
| Conversion Rate | клиент купил рекомендованный продукт |
| Incremental uplift | прирост к контрольной группе |
| Revenue per recommendation | прибыль на одно предложение |
| Complaint rate | доля негативных реакций |

## 6. Когда переобучать модель

Модель стоит переобучать, если:

- упал `MAP@7` на свежей отложенной выборке;
- вырос data drift по ключевым признакам;
- изменилось распределение популярных продуктов;
- появились новые продукты;
- заметно ухудшились бизнес-метрики.

## 7. Реализация в проекте

Prometheus-метрики определены в:

```text
src/bank_recs/monitoring.py
```

Метрики отправляются из:

```text
app/main.py
```

Endpoint для сбора метрик:

```text
GET /metrics
```
